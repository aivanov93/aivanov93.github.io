Listing Management (REST) 

Introduction

The REST Listing Management APIs allow you to create and manage listings.  Managing a listing involves calling multiple APIs
  • Basic listing details, such as name and location using Listings API.
  • Detailed listing description in multiple languages, including space summary, house rules, etc. using Listing Descriptions API.
  • Multiple photos using Photos API.
  • Booking settings, such as check in/check out time, cancellation policy, etc using Booking Settings API. 
  • Availability rules, such as advanced notice, turnover days using Availability Rules API. 
  • Pricing settings, such as cleaning fee, weekly/monthly discounts etc using Pricing Settings API. 
  • Custom pricing and availability for each day using Calendars API or Calendar Batch API.
  • Seasonal rule groups of pricing and availability to specify special pricing or availability adjustments for seasons using Seasonal Rule Group API.
Before proceeding, please ensure OAuth and Webhook Notifications are integrated and working properly. See the Getting Started guide for more information.
To implement REST API Listing Management, you need vr scope, which you should have been granted during the application credentials process as detailed in Getting Started.

Publishing a Listing

Your application can create new listings as well as connect to existing listings that are not API-connected on behalf of a Host. You need to obtain OAuth authorization from the Host in order to create and manage listings on their behalf. See Getting Started for more information.
When publishing a listing from your application, you have the following options:
  •  Create a new listing. You can:
    • Create and automatically publish a new listing.
    • Create a listing and let the Host decide whether to publish it as new or merge it with an existing listing.
  • Connect to an existing listing via the API.

Sync Categories

An Airbnb listing that is synced via your application contains a special field called synchronization_category (or sync category), which describes how the listing is being synced.

Setting of the sync category can be done in two ways:
  • You can set the sync category through the API when publishing a listing from your application. You do so by either creating a brand new listing with a sync category or connecting to an existing listing on Airbnb by writing a sync category to it. 
  • Alternatively, you can leave this action to be done by the Host on the Airbnb sync page.

Creating a New Listing

The steps below describe the flow for publishing a new listing from your application.
  • Create a listing: Call the Listings API to create a listing with basic details. By default, the requested_approval_status_category is set to "new", meaning Airbnb's automated review process is not invoked. When you create a listing, you have two options for the synchronization_category:
    • Set it to "sync_all" to mark it as a new listing to be automatically published.
    • Set it to "sync_undecided" (the default value) to let the Host decide whether to publish it or merge it with an existing listing.
  • Update details: Call other APIs to upload or update listing attributes, photos, descriptions, availability and pricing to match the listing data in your software. Do not update the has_availability field until the listing is published, as setting it to false will reset the approval status to "new".
  • Request review: Once the listing is ready, call the Listings API again to update the listing and set the requested_approval_status_category to "ready for review", triggering Airbnb's automated review. The review validates the listing against the API Program Listing Requirements. If the listing is rejected, you must address the rejection reason(s) and re-submit for review. Please note that some listings require manual approval, in which case the status  will remain as "ready_for_review". Once these listings are reviewed by Airbnb, you will be notified of the result.
  • Publish: Once the listing is approved, one of three things will occur.
    • If the  synchronization_category was set to "sync_all" via the API then it will automatically be published, and has_availability will be set to true.
    • If the synchronization_category is "sync_undecided" and the Host has no existing listings on Airbnb that are not API-connected, then the new listing will automatically be published. The sync category will be set to Everything (i.e. "sync_all") and you will receive a Listing Synchronization Settings Updated notification. The Host can subsequently change the sync category at any time on the sync page.
    • If the synchronization_category is "sync_undecided" and the Host has existing listings on Airbnb that are not API-connected, the Host must go to the sync page to ensure a duplicate listing is not created when publishing.
You cannot create a new listing with a Limited sync category. The Host must first create the listing on the Airbnb website and then your application can connect to it with a Limited sync category.

Merging with an Existing Listing on the Sync Page

If you create a new listing without setting the sync category via the API and the Host has existing listings on Airbnb that are not API-connected, the Host must go to the Airbnb sync page to choose whether to merge the newly-created listing with an existing listing, or publish it as a new listing.
On the sync page the Host can:
  • merge a newly-created API-connected listing with an existing listing on Airbnb, if applicable. Note that the host can only merge with existing listings that they are an owner of,  cohosted listings are not supported.
  • choose the sync category for the newly-created API-connected listing and publish it.
You will receive a Listing Synchronization Settings Updated notification whenever the Host performs an action on the sync page. See this Help Center article for more information on merging listings using the sync page.
The example below describes what happens when a Host merges a newly-created API-connected listing with an existing listing.
  • Your application successfully creates a surrogate listing through the API which gets assigned a listing ID of 12345.
  • On the sync page, the Host merges this listing with one of their existing listings with a listing ID of 67890.
  • You receive a Synchronization Settings Changed notification that contains both IDs.
  • Your application is now allowed to edit listing 67890. See Existing Listing Content for details on what happens when the existing listing becomes API-connected.
  • Listing 12345 is deleted.

Connecting to an Existing Listing via the API

You can connect to an existing listing that is not API-connected when publishing from your application through the API. To do so, first retrieve the Host's existing listings and provide an interface for the Host to map the listing in your software to an existing listing on Airbnb. Then call the Listings API to write a sync category to the existing listing. 
You can only connect to existing listings owned by the Host. You cannot connect to listings that are cohosted by the Host.

Retrieving a Host's Existing Listings

You can fetch the Host's existing Airbnb listings using the Listings API.
  • To retrieve only the listings that are not API-connected, select the listings whose synchronization_category field is null.
  • To retrieve only the listings that the Host owns (i.e. is not a cohost of), use the exclude_cohosted_listings query parameter.

Connection Steps

The steps below describe the flow for connecting to an existing listing from your application.
  • Connect to an existing listing: Call the Listings API to update the listing and set the synchronization_category to either "sync_all" or "sync_rates_and_availability". Unless specified in the request, the requested_approval_status_category will be initialized to "new" and the listing will be temporarily unlisted until it is approved.  See Existing Listing Content for details on what happens when an existing listing becomes API-connected.
  • Update details: Call other APIs to upload or update listing attributes, photos, descriptions, availability and pricing to match the listing data in your software. Do not update the has_availability field until the listing is published, as setting it to false will reset the approval status to "new".
  • Request review: Once the listing is ready, call the Listings API again to update the listing and set the requested_approval_status_category to "ready for review", triggering Airbnb's automated review. The review validates the listing against the API Program Listing Requirements. If the listing is rejected, you must address the rejection reason(s) and re-submit for review.
  • Publish: Once the listing is approved, it will automatically be published, i.e. has_availability will be set to true.
Note: If you use "sync_rates_and_availability" and the existing listing's content does not meet the API Program Listing Requirements, the connection will be rejected. The existing listing will be unaffected by the connection attempt.

Existing Listing Content

When you connect to an existing listing on Airbnb via the API, or when the host merges a newly-created API-connected listing with an existing listing on the sync page, none of the existing listing's content is replaced. All reservations, reviews, and content of the existing listing remain untouched. Most pricing and availability settings also remain, with the following exceptions:
  • Any imported iCals on the existing listing are unlinked when it becomes API-connected. This may open up dates that were previously blocked by iCals. After the connection is made, check the listing's availability and make necessary updates through API.
  • The existing listing is converted to Instant Book if it was previously Request to Book.
  • Custom pricing and availability settings, such as discounts, seasonal rules and standard fees are cleared as they may not be supported by your application. After the connection is made, perform any updates that are needed.
To update the existing listing's information, make all the necessary listing update calls. 
You will not receive any webhook notifications for existing bookings for the newly connected listings. We suggest you perform an initial fetch of all existing reservations using the Reservations API.

Syncing Only Pricing and Availability

If you only have pricing and availability data for a listing and want to sync that with Airbnb without providing listing content, you can do so using Limited sync.
  • First, the Host must create the listing on the Airbnb website if it is not already created. They can enter any valid values for pricing information as these fields will be overwritten when your application connects to the listing. An API-connected Host can create listings on the Airbnb website.
  • Once the listing has been created, you can connect to the listing and set the sync category to "sync_rates_and_availability". You do not need to provide any content when you connect to the listing.
  • The listing is now connected with your application and will sync pricing and availability data from it. Any pricing data on the listing prior to the connection is overwritten with data from your application.

Managing a Published Listing

Updating the Sync Category

You can update the sync category of a listing at any time. The Host can also update the sync category of a listing after it is published via the sync page. You will receive a Listing Synchronization Settings Updated notification when any changes to the sync category occur. 

Updating Limited Sync Listings

For listings with a Limited sync category, API requests to update listing content are blocked and will return a 403 (Forbidden) error. 
Your application has full write access to the  /pricing_settings, /availability_rules,  /seasonal_rule_sets, /seasonal_rule_set_timelines, /calendars,  and /calendar_operations endpoints. For the /listings endpoint, however, only the has_availability , total_inventory_count and property_external_id fields can be updated for Limited sync listings.

Unlisting Listings

To unlist a listing, set its has_availability field to false. This removes the listing from search results and hides it from guests' view. While unlisted, you can still update the listing's content. To re-list the listing, set the has_availability field to true. 

Disconnecting (Unlinking) Listings

An API-connected listing can be disconnected from your application at any time. The Host can do this on the sync page using the “Stop Sync” button.  Your application can do this by calling the Listings API to update the synchronization_category to null. Once disconnected, the Host must manage the listing directly on the Airbnb website. You will receive a Listings Unlinked notification when a listing is disconnected from your application. Once disconnected, any updates to that listing through the API will result in a 403 error.
The Host cannot reverse a stop sync. If the Host wants to resume managing the listing through your application, you must connect to the listing again.

Deleting Listings

You can choose to completely delete a listing. See Deleting a Listing. You will receive a Listings Unlinked notification when this happens.

Disconnecting the Host's Account from your Application

Your application can be disconnected from the Host's account at any time. You can do this by revoking their token and the Host can perform this action from their Account Settings. When this occurs, all the Host's listings will be disconnected from your application and you will lose access to the Host's account. You will receive an Authorization Revoked notification when a Host's account is disconnected from your application.

API Program Listing Requirements  

All listings published via the API are subject to our Listing API Program Policies. Any listing that fails to meet the requirements of the policies will not be approved for publishing.

Notes on Testing

Creating Test Listings

Do not create test listings under the Airbnb partner account (i.e. the account that owns the test application). Instead, create a new Airbnb account and authorize your test application for access.

Always authorize your test application with your test user account before creating test listings with that account. Once you authorize your test application with the account, any listings created with that account will only be visible to your test users. Furthermore, these listings will not be subject to our fake inventory monitors. Creating test listings without first authorizing your test application with the test user account could result in those listings being deleted by our fake inventory monitors. 

Testing Checklist

Before granting supply partners access to a production application, Airbnb will verify that they have completed the following listing management checklist using their test application.

Onboarding listings

  • Create listings that both meet and don’t meet our minimum standards.
  • Confirm receipt of listing approval notifications (both approval and rejections).
  • Publish at least 5 listings. Make sure to test all variations:
    • publishing a new listing with full sync and with with limited sync
    • connecting to an existing listing with full and with limited sync. When connecting to existing listings make sure to import all current reservations. Tip: a quick way to create listings to use for connecting to is to stop sync some API listings, which will disconnect them from your application.
  • Confirm the receipt of the Synchronization Settings notifications.

Managing Listings

  • Make sure you can successfully unlist and relist listings.
  • Try unlinking (stop sync) several listings. Confirm the receipt of unlinked notifications.
  • Try unlinking the entire account in two ways: by revoking a token, and by clicking Disconnect under Connected Apps as a host. Confirm the receipt of the authorization revoked notification.

Rate Limits

Please first review the general API Rate Limiting requirements and ways to handle such issues. Listing Management comes with its own specific requirements, described below.
  • POST/PUT .../listings . We have a per application limit of 40 lat, lng updates per minute. Once reached, an error is returned for up to one minute.
  • POST .../listing_photos . We have a per application limit 50,000 image uploads per hour. Once reached, an error is returned for up to one hour.
  • POST/PUT .../calendar_operations requests. We have a per application limit of 160 batch calendar uploads per minute. Once reached, an error is returned for up to one minute.

Seasonal Rule Group API

You can use Seasonal Rule Group objects to create pricing and availability rule sets and then apply them to listings for a specified date/time period. If you prefer to send us your pricing and availability rules for specific periods of time as a flat set of records, use the LOS Records API. The LOS Records API offers more flexibility than Seasonal Rule Groups.

A Seasonal Rule Group is comprised of Pricing Rule objects and one Availability Rule object. You can create up to 366 Seasonal Rule Groups and up to 3000 individual Rules in total.
Implementing Seasonal Rule Groups is a two-step process:
  • Create Seasonal Rule Groups as standalone objects
  • Apply Rule Groups to listings for specific date ranges. See Listing Rule Group API.

Pricing Rules Overview

With Pricing Rules, you can set the following types of pricing adjustment or discount rules to be applied to your highest precedence daily rate (i.e rules are applied on top of custom pricing). Note that an “adjustment” can be either positive or negative.
  • Seasonal adjustment - A percentage or fixed dollar amount applied to the base nightly price (daily or custom price). For example, if they book during Christmas week when demand is high, they have to pay 40 USD more per night. This new nightly price is referred to as the adjusted seasonal price.
  • Length-of-stay adjustment - A percentage or fixed dollar amount applied to the adjusted seasonal price if they stay a specified length of time. Overwrites weekly/monthly discounts. For example, if they stay more than 7 nights, they pay 10% more. If the adjusted seasonal price is $100, then the new nightly price is $110.
  • Last-minute discount - Modify the adjusted seasonal price  by a percentage or fixed dollar amount if guests book within a specified date/time of their stay. For example, if they book a week or less before their stay, they get a 10% nightly discount. If the adjusted seasonal price is $100, then the new nightly price is $90.
  • Booking-ahead discount - Deduct a percentage or fixed dollar amount from the adjusted seasonal price if they book before a specified date/time before their stay. For example, if they book 60 days  before their stay, they get a $20 nightly discount. If the adjusted seasonal price is $100, then the new nightly price is $80. 
You can set price adjustments in one of two ways:
  • Absolute - A fixed dollar amount, e.g., $20
  • Percent - A percentage, e.g., 10%
For examples and more details on how pricing rules are applied see the appendix section about Applying Pricing Rules.

Availability Rules Overview

We support the following types of availability rules: 
  • Days of the week (Mon-Sun) when guests cannot check-in
  • Days of the week (Mon-Sun) when guests cannot check-out
  • For each day of the week (Mon-Sun), the maximum nights guests can stay if they check-in on that day
  • For each day of the week (Mon-Sun), the minimum nights guest have to stay if they check-in on that day
Custom daily changeover values marking the dates as unavailable for checkin/checkout overwrite seasonal default/availability rules for checkin and checkout, if the latter mark days as available. Custom daily changeover values marking the dates as available for checkin/checkout don't overwrite seasonal/default availability rules if the latter mark days as unavailable.
For more details on how availability rules are applied see the appendix section about Applying Availability Rules.

LOS Records API

This API allows you to set a listing's pricing and availability for specified date/time periods by using a list of length-of-stay records (LOS records). It is an alternative to the Seasonal Rule Group API that allows for more flexibility in setting pricing and availability for specific dates and stays.

Applying or Removing LOS Records

This action lets you apply or remove LOS records to/from a listing for a given set of dates.
text

To apply LOS records:
  • Call /v2/los_records with a non-empty los_records array in the payload.
To remove LOS records:
  • Call /v2/los_records with an empty los_records array AND do a full update.
When applying LOS records, you must also provide a base nightly rate for the listing using the Pricing API. This value will only be used to display the price when guests search without dates and does not impact availability.
For more details on how applying LOS records interacts with other pricing and availability rules, see the appendix section about Applying LOS Records.

LOS Record Data Format

The general format for each string in the los_records array is:
text
<checkin date>,<max guest number>,<1 night price>,<2 nights price>,<3 nights price>,...

Example

text
2019-01-01,1,100,210,0,390,550,580,0,820
If 1 guest checks in on 2019-01-01, the price for staying 1 night would be 100, the price for 2 nights would be 210, and staying 3 nights is not available.
text
2019-01-01,2,200,410,0,690,1050,1080,0,1620
Same as above, but this record indicates the prices for 2 guests checking in on 2019-01-01.
text
2019-01-01,4,400,810,0,1290,1550,1580,0,1820
Same as above, but this record indicates the prices for >2 and <=4 guests.
text
2019-01-01,5,0
The listing is not available for 5 guests if check-in date is 2019-01-01.
text
2019-01-01,6,600,1620,2350,3500,0,4230,4750,5240
Prices for 6 guests if check-in date is 2019-01-01.
text
2019-01-02,1,100,210,0,390,550,580,0,820
Prices for 1 guest if check-in date is 2019-01-02.
text
2019-01-02,4,300,610,0,990,1550,1580,0,2020
Prices for >1 and <=4 guests if check-in date is 2019-01-02.

Note that:
  • The check-in date should be within the next 18 months (550 days). If the LOS records do not contain prices for certain check-in dates, we will mark these dates as not available for check-in.
  • For any combination of check-in date and max guest number, we support up to 28 days' length-of-stay prices. This is a maximum cap; you can send prices for >=1 and <=28 nights. If the LOS records do not contain prices for certain length-of-stays, we will mark these dates as not available for check-in. For stays longer than 28 days, we will mark days as unavailable.
  • If the LOS records do not contain data for certain dates, either as check-in dates or as part of a stay, we will mark these dates as unavailable.
  • Each price should be an integer. If the listing is not available for a particular length-of-stay, the price should be set to 0.
  • The guest number is the max guest number (please see the example on third row). This number should be an integer >=1 and <=16, as bookings on Airbnb usually do not differentiate the prices beyond 16 guests.

Listing Regulations API

This API lets you submit required listing regulation information for an Airbnb listing in markets where such information is required by local regulations. To submit listing regulation information, you must provide one or more of a listing's existing registration with the market's regulatory body, a valid exemption from listing registration, or a listing residency category. Supported markets are listed in the sections below.

For these listings, partners can provide Airbnb with a listing's total inventory count, daily available count for specific dates, as well as property external id. Note that the property type house and deprecated fields are not eligible for Room Type Inventory. You must add the property_type_category to use Room Type Inventory.

Standard Fees

Partners can provide Standard fees that will apply to reservations on top of the stay price. Fees can be specified as either percentage of the nightly price (after all adjustments) or a flat amount per reservation.

Nightly price

In order to streamline the guest experience, the fees will not be shown as separate line items. Our system performs the calculation for the fee and  they are included in the nightly rate or cleaning fee when shown to guests. However,  Airbnb provide a detailed breakdown of these fees in reservation webhook notifications (see Reservation Object), so Airbnb transactions will be more consistent with a property manager's existing accounting structure. Airbnb currently supports 4 types of fees.
Fees are subject to the same taxes/fees and cancellation/refund policies as the price type they're mapped to. All applicable taxes and the Airbnb host and guest fees will be applied. Fees will inherit the cancellation/refund policy of the price type they're mapped to. For example, cleaning fees are refunded if a guest cancels before the reservation starts; this will apply to the linen fee as well. Similarly, if a strict cancellation policy refunds 50% of the stay price, we will refund 50% of any resort fees charged. 
If you have previously been manually customizing the base price to include any of the above fee types, remember to reduce the base price by the relevant amount after developing to standard fees. This ensures that you are not “double-charging” the fee.

Applying Pricing Rules

Setting an Adjusted Seasonal Price

You can apply a percentage or fixed dollar amount change to the nightly price of highest precedence (base or custom) to get the nightly adjusted seasonal price. This can add to, or subtract from, the base nightly price.
Example: This rule adjusts the base nightly price 20% higher. If the default nightly price is $100 USD, then the adjusted seasonal price is $120 USD.
json
{
  "type": "SEASONAL_ADJUSTMENT",
  "priceChange": {
    "amount": 20,
    "type": "PERCENT"
  }
}

Setting a Length-of-Stay Adjustment

If a stay is  X days or more, you can increase or decrease the adjusted seasonal price a percentage or fixed dollar amount. This is similar to existing long-term adjustments applied to trips that are at least 7 days and 28 days respectively. This rule is applied to the longest day matched. This discount is shown to guests.
Example: This rule says this if the guest stays at least 3 nights, then they get a 20% nightly discount. If the adjusted seasonal price is $100 USD, then the new discounted price is $80 USD.
json
{
  "type": "STAYED_AT_LEAST_X_DAYS",
  "threshold": 3,
  "priceChange": {
    "amount": -20,
    "type": "PERCENT"
  }
}

Setting a Last-Minute Discount

You can offer percentage or fixed dollar amount discounts to trips that are booked X days or less before a stay. This discount is not shown to guests.
Example: If the guest books 3 days or less before their stay, this rule entitles them to a $20 nightly discount. If the adjusted seasonal price is $100 USD, then the new discounted price is $80 USD.
json
{
  "type": "BOOKED_WITHIN_AT_MOST_X_DAYS",
  "threshold": 3,
  "priceChange": {
    "amount": -20,
    "type": "ABSOLUTE"
  }
}

Setting a Booking-Ahead Discount

You can offer percentage or fixed dollar amount discounts to trips that are booked X days or more in advance. This discount is not shown to guests.
Example: If the guest books 60 nights or more in advance, this rule entitles them to  $20 per night discount. If the adjusted seasonal price is $120 USD, then the new discounted price is $100 USD.
json
{
  "type": "BOOKED_BEYOND_AT_LEAST_X_DAYS",
  "threshold": 60,
  "priceChange": {
    "amount": -20,
    "type": "ABSOLUTE"
  }
}

Calculating the Final Price

Adjusted seasonal price = Seasonal price adjustment applied to base nightly price.
Final price = Additional price adjustments applied to the adjusted seasonal price.
Suppose for a certain date range, the following pricing rules are created:
Base nightly price = $200
  • Seasonal adjusted discount = -10%,  nightly discount = -(200 * .10) = -20
Adjusted seasonal price = 200 - 20 = $180
  • Long-term adjustment = -20, nightly discount = -20
  • Last-minute discount = -2%,   nightly discount = -(180 * .02 ) = -3.6
  • Booking-ahead discount = -2%,  nightly discount = -(180 * .02) = -3.6
Final price  = 180 - 20 - 3.6 - 3.6 = $152.8

Staying Across Seasons

Pricing rules are resolved on a per day basis. For a listing with base price $100, suppose there is 5% discount from 5/1 - 5/5, and $10 discount from 5/8 - 5/10. If the stay is from 5/1 - 5/10, it will get 5% off from 5/1 - 5/5, no discounts from 5/6 - 5/7, and $10 off for each day from 5/8 - 5/10. Thus, the total cost of the stay is 100 * (1-0.05) * 5 + 100 * 2 + (100 - 10) * 3 = 945.

Applying Availability Rules

Setting When Guests Cannot Check-in

You can set which days of the week guests cannot check-in.
Example: The following payload indicates that guests cannot check-in on Monday and Friday:
json
{
  "closed_for_checkin": ["MONDAY", "FRIDAY"]
}

Setting When Guests Cannot Check-out

You can set which days of the week guests  cannot check-out.
Example: The following payload indicates that guests cannot check-out on Sunday:
json
{
  "closed_for_checkout": ["SUNDAY"]
}

Setting the Minimum Nights Guests Have to Stay 

You can set the minimum number of nights guests have to stay depending on which day of the week they check-in.
Example: The following payload sets two rules: 
  • If the guest checks-in on Sunday, Tuesday, Friday or Saturday, they have to stay for 5 nights minimum. 
  • For all other days, the guest has to stay for 3 nights minimum. UNSPECIFIED indicates that this applies to all days that haven't had a value specifically set. 
json
{
  "min_nights": {
    "SUNDAY": 5,
    "TUESDAY": 5,
    "FRIDAY": 5,
    "SATURDAY": 5,
    "UNSPECIFIED": 3
  }
}

Setting the Maximum Nights Guests can Stay 

You can set the maximum number of nights guests can stay depending on which day of the week they check-in.
Example: The following payload sets two rules: 
  • If the guest checks-in on Monday, Wednesday or Thursday,  they can stay for 8 nights maximum. 
  • For all other days, the guest can stay for 6 nights minimum. UNSPECIFIED indicates that this applies to all days that haven't had a value specifically set. 
json
{
  "max_nights": {
    "MONDAY": 8,
    "WEDNESDAY": 8,
    "THURSDAY": 8,
    "UNSPECIFIED": 6
  }
}

If you use LOS records to set a listing's price and availability, some of the parameters in the Pricing Settings API, Availability Rules API and Calendar Settings API must be omitted. Interaction with the Availability Rules API
We will continue supporting all existing availability rules for listings using LOS records. 
For any combination of [check-in date, guest number, length of stay], if the LOS record price is 0 then we will mark it as unavailable, and all the other Availability Rules will not be able to change it. However, if the LOS record shows this date as available, the other Availability Rules will be applied on top. This may add additional restrictions to the date and possibly render it unavailable for certain stays.
If a date is not included in any LOS record, either as a date available for check-in or as a date available for a stay, we will mark it as unavailable.

Interaction with Nightly Prices, Standard Fees and Discounts

  • Stay price adjustments for seasonal pricing, length of stay discounts, or any other price adjustment that are not supported by Seasonal Rule Groups can be incorporated into LOS records.
  • Standard fees should be sent separately so they can be itemized during the guest checkout flow.
  • Discounts that need to be displayed separately to guests should be sent separately and will be applied on top of the LOS record prices.

FAQ

Why does the Airbnb calendar display dates as blocked when our app set them as available?

If a Host cancels or initiates alterations to a reservation’s start/end dates, the original dates stay blocked.
